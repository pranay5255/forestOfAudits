Research operating model

The artifact should explain the whole loop.

A useful open artifact does not stop at a dataset upload. It documents how tasks were exposed, how labels were hidden, how rollouts were generated, how grading happened, and how downstream training was evaluated.

1

Task pool

SWE-rebench V2-PRs provides issue/PR tasks. EVMBench provides audits, modes, source snapshots, and submission contracts.

2

Generation

Agents inspect source and submit artifacts without Docker, tests, gold findings, hidden exploits, or grader feedback.

3

Labeling

The completed rollout is replayed or graded with the real execution environment. Labels are attached after the fact.

4

Training data

Successful traces feed SFT. Losing branches become preference pairs, judges, rerankers, or value-model targets.

5

Held-out eval

A fixed, leakage-free EVMBench split measures whether the model actually improves with data and compute.

Core thesis Execution-free generation gives scale. Execution-backed labeling gives truth. The research artifact is the bridge that makes both inspectable.

Marin issue discourse

Read the issues as a research program, not a flat list.

The strongest structure is chronological and conceptual: survey, generation recipe, scale, execution validation, SFT validation, and small-model failure analysis.

1. Context and motivation

#3093 Agentic SFT dataset tracker

Surveys reusable agentic datasets and exposes the recurring artifact problems: mixed licenses, unclear scaffold formats, weak correctness labels, and unclear split hygiene.

#4435 Generate SWE-ZERO-style traces

Frames the central need: Dockerized runnable environments are expensive and scarce, while execution-free traces can scale across more repositories and scaffolds.

2. Execution-free generation recipe

#4561 SWE-ZERO MVP

The practical recipe: mini-swe-agent format, real worktrees, PATH-whitelisted bash, 32K context, no tests, no Docker, no runtimes. This is the direct EVMBench generation template.

#5588 Reproduce mini-coder-1.7B

Anchors the teacher/harness baseline. EVMBench should do the same for whichever model generates initial audit, patch, or exploit trajectories.

3. Scaling path and SWE-rebench V2 placement

#4653 Multi-language scaling

Shows the recipe transfers beyond Python, but correctness drops. Lesson: grounded traces can still be useful while final submissions remain noisy labels.

#4666 1B-token scaling

Scales to roughly 96K rollouts from 32K PRs x 3. This is the first proof that the pipeline can produce training scale, not just demos.

#4710 V2-PRs pilot

Moves from container-rich V2 to broader PR-only V2-PRs. The point: execution-free generation needs problem, source, base commit, and patch/task metadata, not images.

#4719 140B-token scale run

Targets roughly 122K PRs x 100 rollouts. Useful for EVMBench because it treats many rollouts per task as a training-data generator and a pass@k research instrument.

4. Execution-backed validation

#4683 ConTree execution evaluation

Replays commands, recovers a patch, applies it in an execution image, and runs tests. EVMBench should mirror this pattern with its Docker grader after generation.

#4383 Python execution traces

A contrast case: runtime execution traces from test runs. Useful if EVMBench later wants EVM call traces as a separate artifact from agentic rollouts.

5. SFT validation and small-model pitfalls

#4898 Marin-8B SWE-ZERO SFT

Shows measurable gains from execution-free trajectories: Marin-8B base went from 0% to the 3-5% range on the 100-task SWE-bench Verified slice.

#5611 Qwen3-1.7B and 8K truncation

Critical warning for EVMBench SFT: truncation, EOS, template mismatches, max tokens, and prompt-conditional degeneration can dominate the result.

6. Adjacent issues for artifact quality

#4963 Agent trace PPL evals

Proxy metrics for next action, tool call, bash command, and patch likelihood.

#4389 Soft proxy for agentic benchmarks

Important caution: likelihood proxies are plausible but noisy and need hard-eval calibration.

#5401 Agentic-coding BPB eval

Shows how success-only trace likelihood can support checkpoint confidence portfolios.

#5095 Diff and patch PPL evals

Relevant for EVMBench patch-mode and exploit artifact distributions.

EVMBench fit

What changes when the domain is smart contracts?

EVMBench has a sharper domain than general SWE tasks. That makes its trajectories valuable, but also increases the cost of leakage and incorrect labels.

Domain signal

Audit reasoning

Teach asset-flow tracing, authorization checks, invariant reasoning, oracle manipulation, accounting drift, and reentrancy reasoning in a way generic SWE data rarely does.

Execution labels

Delayed grader truth

The model should not see the grader while acting. The grader should label outcomes later: detected findings, exploit success, patch safety, terminal score, and failure mode.

Forest data

Branch comparisons

Forest-of-Thought runs naturally create winners and losers. Those branches can become preference pairs, judge data, reranker data, and value-model labels.

Guardrails

The artifact should make failure visible.

These are the mistakes most likely to produce misleading conclusions. They should be tracked as first-class metrics.

Metric trap

Submitted is not correct

Track generated, submitted, structurally valid, graded, successful, and high-confidence successful as separate counts.

Leakage

Gold files must be excluded

Hide gold findings, exploit scripts, oracle patches, grader logs, hidden tests, and any task docs that reveal the answer.

Training stability

Templates and EOS matter

SWE-ZERO showed that truncation, packing, stop tokens, and prompt formatting can cause catastrophic degeneration.

Release checklist

What to publish so others can build on the work.

This checklist is the difference between a result screenshot and a reusable open-source research artifact.

Dataset release
  • Task manifest with audit id, mode, source snapshot, split, and hidden-artifact policy.
  • Raw trajectories exactly as produced by the agent harness.
  • Normalized decision-point rows for action-level research.
  • Branch summaries with terminal score, success, failure reason, and artifact paths.
  • Preference pairs from forest branches and repeated rollouts.
  • SFT exports with the exact chat template and truncation policy.
  • Filtering report listing dropped examples and reasons.
Experiment release
  • Generation prompt, allowed commands, blocked commands, and no-Docker/no-test guarantee.
  • Model, tokenizer, temperature, context length, max turns, seed, and stop tokens.
  • Post-hoc grading command, Docker image version, and grader commit.
  • Held-out eval task list and per-task score table.
  • Training hyperparameters, packing, LR, batch size, and epoch count.
  • Degeneration audit, invalid-output audit, and leakage audit.
  • Dataset card and model card with limitations and intended use.
Final standard for the artifact A researcher should be able to regenerate the trajectories, rebuild the SFT dataset, rerun the held-out EVMBench eval, and understand why each failure mode was or was not filtered.
