# Test-Time Scaling Strategy for EVMBench with mini-swe-agent and Modal

## Overview

This document replaces the old Yudai-specific plan with a `mini-swe-agent`-first strategy.

The target state is:

- EVMBench remains the outer benchmark system.
- `evmbench/agents/mini-swe-agent/` is the benchmark-facing agent family.
- The local single-agent path uses the `mini` CLI in YOLO mode.
- The Modal/TTS path composes `mini-swe-agent` programmatically through `DefaultAgent` and `SwerexModalEnvironment`.

## Architecture Principles

- EVMBench owns task setup, audit data, mode-specific instructions, RPC wiring, and grading.
- `mini-swe-agent` owns the agent loop, model interaction, trajectory output, and shell execution semantics.
- Modal is the infrastructure layer for remote and parallel execution.
- The OpenAI API key is the provider credential for both local and Modal paths.
- EVMBench submission artifacts remain the grading source of truth.

## Why Switch from the Custom Yudai Loop

The old Yudai agent embedded a custom Anthropic loop inside `start.sh`.

The new plan is to stop carrying a bespoke loop and instead use:

- the documented `mini` CLI for the local benchmark-facing agent
- the documented `DefaultAgent` for programmatic orchestration
- the documented `SwerexModalEnvironment` for Modal execution

This gives us:

- a maintained agent runtime instead of benchmark-local control flow
- native trajectory output
- a clean path from local baseline to Modal TTS
- easier compatibility with OpenAI-backed models

## Benchmark Contract

The current EVMBench solver stages:

- `start.sh`
- one rendered instruction file such as `AGENTS.md`

It does not automatically upload extra runtime config files from the agent folder.

That constraint drives the first integration design:

1. EVMBench renders the mode-specific instructions into `/home/agent/AGENTS.md`.
2. EVMBench uploads `/home/agent/start.sh`.
3. `start.sh` writes a small `mini` override config into `$AGENT_DIR`.
4. `start.sh` invokes `mini -y --exit-immediately -c mini.yaml -c "$AGENT_DIR/mini-override.yaml"`.
5. EVMBench grades only the usual submission artifacts.

## New Agent Folder

The benchmark-facing folder is:

- `evmbench/agents/mini-swe-agent/config.yaml`
- `evmbench/agents/mini-swe-agent/start.sh`

Recommended initial variant:

- `mini-swe-agent-default`

Recommended config shape:

```yaml
env-config-vars: &env-config-vars
  OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
  MODEL: openai/gpt-5
  STEP_LIMIT: "50"
  COST_LIMIT: "20.0"

common: &common_settings
  start: mini-swe-agent/start.sh
  instruction_file_name: AGENTS.md
  gateway_sni_hosts:
    - api.openai.com

mini-swe-agent-default:
  <<: *common_settings
  env_vars:
    <<: *env-config-vars
```

## Local EVMBench Flow

The baseline local path should be intentionally thin:

1. Read env from EVMBench:
   - `OPENAI_API_KEY`
   - `MODEL`
   - `STEP_LIMIT`
   - `COST_LIMIT`
2. Generate a small override config that:
   - sets `agent.mode: yolo`
   - sets `agent.confirm_exit: false`
   - points `environment.cwd` at `/home/agent/audit`
   - sets the OpenAI model via LiteLLM
3. Invoke:

```bash
mini -y --exit-immediately \
  -c mini.yaml \
  -c "$AGENT_DIR/mini-override.yaml" \
  -t "$PROMPT" \
  -o "$LOGS_DIR/mini-swe-agent.traj.json"
```

4. Let EVMBench keep grading:
   - `submission/audit.md` for detect
   - repo diff for patch
   - `submission/txs.json` for exploit

The trajectory file is auxiliary debugging data, not the benchmark submission.

## Config Split

Use two layers of config:

- built-in `mini.yaml` from the package
- a repo-owned EVMBench override generated by `start.sh`

This is better than depending on `mini-extra config` because benchmark runs should not rely on mutable user-global state.

The override file should only carry EVMBench-specific behavior:

```yaml
agent:
  step_limit: 50
  cost_limit: 20.0
  mode: yolo
  confirm_exit: false
  system_template: |
    You are an expert smart contract auditor and developer working inside EVMBench.
  instance_template: |
    First read /home/agent/AGENTS.md for your detailed instructions.
    Then solve the task autonomously inside /home/agent/audit.
    Write any required final artifacts to /home/agent/submission.

    Task:
    {{task}}
environment:
  cwd: /home/agent/audit
  timeout: 240
model:
  model_class: litellm
  model_name: openai/gpt-5
```

## Modal Strategy

For Modal-backed TTS, do not shell out to the local `mini` CLI as the core orchestration layer.

Use:

- `DefaultAgent` for programmatic control
- `SwerexModalEnvironment` for remote execution

This matches the official `mini-swe-agent` runtime split and makes scout/branch/judge composition much easier than wrapping the CLI.

### Modal Config Shape

The Modal worker config should look like this conceptually:

```yaml
agent:
  agent_class: default
  system_template: |
    You are an expert smart contract auditor and developer working inside EVMBench.
  instance_template: |
    First read /home/agent/AGENTS.md for your detailed instructions.
    Then solve the task autonomously.

    Task:
    {{task}}
  step_limit: 50
  cost_limit: 20.0
environment:
  environment_class: swerex_modal
  image: "<evmbench audit image>"
  cwd: /home/agent/audit
  timeout: 240
  startup_timeout: 600
  runtime_timeout: 3600
  deployment_timeout: 3600
  env:
    OPENAI_API_KEY: "${OPENAI_API_KEY}"
model:
  model_class: litellm
  model_name: openai/gpt-5
```

### EVMBench Mapping

Map EVMBench inputs into the Modal worker as follows:

- EVMBench `task.docker_image` -> `SwerexModalEnvironment.image`
- EVMBench audit contents -> uploaded or mounted under `/home/agent/audit`
- EVMBench rendered instructions -> `/home/agent/AGENTS.md`
- EVMBench output contract -> unchanged

## Forest-of-Auditors Plan

The TTS path should still use the same high-level flow:

1. scout pass
2. activate 2-4 specialist trees
3. run 2 or more branches per tree
4. tree-local merge
5. global judge and final report

Suggested initial tree set:

- `token-flow`
- `accounting`
- `access-control`
- `cross-contract`
- `exploitability`

Each worker remains a `mini-swe-agent` worker with a role-specific prompt supplement.

## Output and Logging

Keep these artifact classes separate:

- EVMBench submissions:
  - `/home/agent/submission/audit.md`
  - patch diff derived by EVMBench
  - `/home/agent/submission/txs.json`
- `mini-swe-agent` trajectories:
  - `/home/logs/mini-swe-agent.traj.json`
  - per-branch trajectories for Modal/TTS
- orchestration metadata:
  - tree selection
  - per-branch budget
  - merge provenance

## Risks and Constraints

- The local benchmark path requires the `mini` CLI to be available in the runtime image.
- If we later want a checked-in repo `mini.yaml` to be uploaded directly, solver changes are required.
- `mini` trajectories are useful for debugging but should not be confused with benchmark submissions.
- Detect should be stabilized first before broadening to patch and exploit.

## Success Criteria

- EVMBench can discover and run `mini-swe-agent-default`.
- One detect audit completes locally with unchanged grading.
- One Modal baseline run completes using `DefaultAgent` plus `SwerexModalEnvironment`.
- Forest mode improves results at matched budget, not just by increasing spend.
## Real Detect Smoke

For a fast end-to-end smoke on a real detect audit, use a 10-step cap and skip
local judge grading:

```bash
set -a
. ./.env
set +a

uv run python evmbench/agents/mini-swe-agent/modal_baseline.py \
  --audit-id 2024-01-canto \
  --image ghcr.io/pranay5255/evmbench-audit:2024-01-canto \
  --step-limit 10 \
  --no-grade
```

For the benchmark-facing local agent config, use `mini-swe-agent-smoke-10`.

## Phase 5 Solver-Native Smokes

Phase 5 exposes Modal through normal EVMBench `agent_id` selection. The local
EVMBench task computer is still used for setup and grading; the Modal runner
receives `MODAL_AUDIT_IMAGE` or `MODAL_AUDIT_IMAGE_REPO` for the registry image
that Modal can pull. The smoke variants set
`MODAL_ALLOW_SMOKE_FALLBACK_SUBMISSION=1`, so a capped run that reaches its
budget before writing `audit.md` still emits a clearly labeled placeholder
report. Production variants should not set that fallback.

Baseline Modal smoke:

```bash
set -a
. ./.env
set +a

export MODAL_AUDIT_IMAGE=ghcr.io/pranay5255/evmbench-audit:2024-01-canto

uv run python -m evmbench.nano.entrypoint \
  evmbench.audit=2024-01-canto \
  evmbench.mode=detect \
  evmbench.apply_gold_solution=False \
  evmbench.log_to_run_dir=True \
  evmbench.solver=evmbench.nano.solver.EVMbenchSolver \
  evmbench.solver.agent_id=mini-swe-agent-modal-baseline-smoke-10 \
  runner.concurrency=1
```

Forest/TTS Modal smoke:

```bash
set -a
. ./.env
set +a

export MODAL_AUDIT_IMAGE=ghcr.io/pranay5255/evmbench-audit:2024-01-canto

uv run python -m evmbench.nano.entrypoint \
  evmbench.audit=2024-01-canto \
  evmbench.mode=detect \
  evmbench.apply_gold_solution=False \
  evmbench.log_to_run_dir=True \
  evmbench.solver=evmbench.nano.solver.EVMbenchSolver \
  evmbench.solver.agent_id=mini-swe-agent-modal-forest-smoke \
  runner.concurrency=1
```

For multi-audit runs, prefer `MODAL_AUDIT_IMAGE_REPO=ghcr.io/pranay5255/evmbench-audit`
so the adapter builds `<repo>:<audit_id>` for each task. The local grading image
must still be available to the EVMBench task computer.
